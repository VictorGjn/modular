# V1.0.4 — Bug Fix Plan & Post-Mortem

## Part A: Post-Mortem — Why These Bugs Were Missed

### The Uncomfortable Truth

Six user-facing bugs shipped in v1.0.0 through v1.0.3 that any human tester would have caught in 30 seconds of clicking around. **674 tests passed** and the product was still broken. Here's why.

### Root Cause 1: Unit Tests ≠ User Tests

Every pipeline stage had unit tests. The `treeAwareRetriever` had tests. The `pipelineChat` had tests. They all tested *their own contract in isolation*. Nobody tested the *assembled product*.

- Unit tests verified: "given these inputs, do I get these outputs?"
- Nobody verified: "when I type a message and press Enter, does the conversation work?"

The test suite created a **false signal of quality**. 674 passing tests meant "the individual bricks are rectangular." It said nothing about whether the house has a roof.

### Root Cause 2: No Browser Smoke Test

Not once during development was a browser opened to manually test the chat flow. The agents wrote code, ran `vitest`, saw green, and shipped. This is the AI-coding equivalent of "it compiled, ship it."

A 30-second smoke test would have caught:
- Bug 3 (Settings crash) — click Settings → MCP tab → crash
- Bug 4 (broken history) — send 2 messages → see nonsensical responses
- Bug 5 (no markdown) — get any response → see raw `**bold**` text
- Bug 6 (no traces) — send a message → no pipeline info visible in chat

### Root Cause 3: `as` Casts Defeat TypeScript

The conversation history bug (Bug 4) is a **type system betrayal**. The history type was correctly defined as `role: 'user' | 'assistant' | 'system'`, but `pipelineChat.ts:216` casts it:

```typescript
...history.map(m => ({ role: m.role as 'system' | 'user', content: m.content }))
```

This `as` cast silently erases the `'assistant'` variant. TypeScript's entire value proposition — catching type errors at compile time — is bypassed by a two-character keyword. The original developer probably copied a type from the `messages` array construction (where only system/user are expected) without thinking about what it means for history that *includes assistant turns*.

**If the code had been written without `as`**, TypeScript would have flagged that `'assistant'` doesn't fit `'system' | 'user'` and the bug would have been caught at compile time.

### Root Cause 4: Parallel Agents, No Integration Review

Multiple sub-agents worked on overlapping files simultaneously:
- One agent rewrote `McpServersTab` to merge `consoleStore` + `mcpStore`
- Another agent was working on the pipeline
- Another was fixing broken flows

Nobody reviewed the *integration*. Each agent's diff looked reasonable in isolation. The `McpServersTab` rewrite introduced an infinite render loop (useEffect depending on a derived array that changes every render), but the agent that wrote it timed out before it could verify its work in a browser.

### Root Cause 5: "Fixed in v1.0.0" ≠ Actually Fixed

Bug 1 (embeddings 400) was supposedly fixed but keeps recurring. This is the classic "I handled the error path I could think of" problem. The fix added filters for empty texts on both client and server. But the 400 can also come from:
- The embedding model not being loaded yet (first request races initialization)
- The embedding service throwing a non-HTTP error that gets caught and re-thrown as 400
- A valid-looking text array that the model rejects (too long, encoding issues)

The fix addressed *one specific failure mode* and declared victory.

### The Meta-Problem

The process was:
1. Spawn agents to write features
2. Agents run unit tests
3. Tests pass → merge
4. Nobody does a manual walkthrough
5. Ship

Missing step: **Someone needs to actually use the product before releasing it.**

### Bug 4: Conversation History Roles (P0)

**File:** `src/services/pipelineChat.ts:216`

**Root cause:** The `as 'system' | 'user'` cast strips `'assistant'` role. All assistant messages are sent to the LLM as `'user'` messages, destroying the turn structure.

**Fix:**
```typescript
// BEFORE (line 216):
...history.map(m => ({ role: m.role as 'system' | 'user', content: m.content })),

// AFTER:
...history.map(m => ({ role: m.role, content: m.content })),
```

The `messages` array type in `executeChat` already accepts `'system' | 'user' | 'assistant'` (see `executionRouter.ts:24`), so removing the cast is type-safe.

**Also fix:** The `executeChat` function signature at `executionRouter.ts:24` explicitly types messages as `{ role: 'system' | 'user' | 'assistant'; content: string }[]` — confirm this is correct (it is).

**Test to add:**
```typescript
// src/services/__tests__/pipelineChat.history.test.ts
it('preserves assistant role in history mapping', () => {
  const history = [
    { role: 'user' as const, content: 'Hello' },
    { role: 'assistant' as const, content: 'Hi there' },
    { role: 'user' as const, content: 'How are you?' },
  ];
  
  // The mapped messages should preserve all roles
  const mapped = history.map(m => ({ role: m.role, content: m.content }));
  expect(mapped[1].role).toBe('assistant'); // NOT 'user'
});
```

**ESLint rule to add** (project-wide prevention):
```json
// In .eslintrc or eslint config
"@typescript-eslint/no-unnecessary-type-assertion": "error"
// And consider: "@typescript-eslint/consistent-type-assertions": ["error", { "assertionStyle": "never" }]
// Or more practically: ban `as` on literal union types
```

### Bug 3: Settings MCP Tab Crash — React Error #185 (P0)

**File:** `src/components/SettingsPage.tsx:718-733`

**Root cause:** Infinite render loop. The `useEffect` depends on `mergedServers`, which is a *new array computed every render* (line 688-701). The effect calls `upsertMcpServer` (line 725), which mutates `consoleStore`, which triggers a re-render, which creates a new `mergedServers`, which re-triggers the effect → infinite loop → React Error #185 (Maximum update depth exceeded).

```typescript
// THE PROBLEM (lines 718-733):
useEffect(() => {
  for (const server of mergedServers) {  // mergedServers is new every render
    upsertMcpServer({                      // this mutates store → re-render
      id: server.id,
      name: server.name,
      description: server.description || server.command,
      connected: server.status === 'connected',
    });
  }
}, [mergedServers, upsertMcpServer]);      // mergedServers changes every render → infinite loop
```

**Fix:** Stabilize the dependency. Use a serialized key (JSON of server IDs + statuses) instead of the array reference:

```typescript
// AFTER:
const mergedServersKey = mergedServers
  .map(s => `${s.id}:${s.status}`)
  .sort()
  .join(',');

useEffect(() => {
  for (const server of mergedServers) {
    upsertMcpServer({
      id: server.id,
      name: server.name,
      description: server.description || server.command,
      connected: server.status === 'connected',
    });
  }
  // eslint-disable-next-line react-hooks/exhaustive-deps
}, [mergedServersKey]);
```

**Alternative (better):** Remove this effect entirely. The sync between mcpStore and consoleStore should happen in the store layer, not in a React component's render cycle. The `loadServers` action in mcpStore should call `upsertMcpServer` directly after fetching.

**Test to add:**
```typescript
// src/components/__tests__/SettingsPage.mcp.test.tsx
it('MCP tab renders without infinite loop', () => {
  // Mock stores with some MCP servers
  // Render McpServersTab
  // Assert it doesn't throw or exceed update depth
  // Assert servers are listed
});
```

### Bug 2: 404 on /api/mcp (P1)

**Investigation findings:**

The server mounts MCP routes correctly at `/api/mcp` in `server/index.ts:70`:
```typescript
app.use('/api/mcp', mcpRoutes);
```

The mcpStore calls `API_BASE` which is `${BASE}/mcp` = `/api/mcp`. This *should* work.

**Likely cause:** The 404 is NOT from the main `/api/mcp` GET endpoint. It's likely from one of these scenarios:

1. **The mcpRoutes module fails to import** — if `server/mcp/manager.js` or any of its dependencies throw during import, the `import mcpRoutes from './routes/mcp.js'` at server startup would fail, and Express wouldn't mount the routes. But this would also break all other routes (unlikely).

2. **A specific sub-route is being called that doesn't exist** — for example, a DELETE or PUT to a server ID that the route doesn't handle (but the router does handle these).

3. **The 404 comes from the SPA fallback** — in production mode, the catch-all `app.get('/{*path}')` serves `index.html` for unmatched routes. If the API routes fail to mount, API calls fall through to the catch-all and get a 200 HTML response (not a 404). So the 404 likely happens in *dev mode* only, where there's no SPA fallback.

4. **Race condition on initial page load** — the mcpStore's `loadServers()` fires before the server is fully ready. The `apiFetch` helper swallows errors and returns `null`, so this wouldn't show as a console 404. Unless there's ANOTHER call somewhere.

**Action items:**
- Add console logging to the server startup to confirm mcpRoutes mount successfully
- Search the *built* JS (not source) for any hardcoded `/api/mcp` calls that bypass the store
- Check browser DevTools Network tab → filter for "mcp" → identify the exact URL and initiator
- Add a catch-all 404 handler in Express (before the SPA fallback) that logs unmatched API routes:

```typescript
// After all API routes, before static/SPA serving:
app.use('/api', (req, res) => {
  console.warn(`[404] Unmatched API route: ${req.method} ${req.originalUrl}`);
  res.status(404).json({ status: 'error', error: `Route not found: ${req.method} ${req.path}` });
});
```

### Bug 1: Embeddings 400 Errors (P1)

**File:** `src/services/treeAwareRetriever.ts:177-209` (client), `server/routes/embeddings.ts:36-71` (server)

**Investigation findings:**

Both client and server already filter empty texts. The 400 isn't coming from empty texts. It's likely one of:

1. **Model not loaded yet** — The embedding service (`server/services/embeddingService.js`) uses `@xenova/transformers` which lazy-loads the model. The first call triggers a download/load. If a second request arrives during loading, it may fail with an unhandled error that the route turns into a 500 (not 400). But the *client* sees it as a non-ok response and throws `Embedding service error: 500`.

2. **Text too long** — MiniLM-L6 has a 512-token input limit. Long code chunks get silently truncated by the tokenizer, but if a chunk is *very* long, the tokenizer itself could OOM or throw.

3. **The real 400 is from a different code path** — `callEmbeddingService` is called from `treeAwareRetrieve` (line 358). But there might be another caller that doesn't filter.

**Fix (defense in depth):**

```typescript
// server/routes/embeddings.ts — add text length limit
// After the existing validation (around line 55):
const MAX_TEXT_LENGTH = 8192; // characters
const truncatedTexts = validTexts.map(t => 
  t.length > MAX_TEXT_LENGTH ? t.slice(0, MAX_TEXT_LENGTH) : t
);
const validEmbeddings = await embeddingService.embedBatch(truncatedTexts);
```

```typescript
// server/routes/embeddings.ts — add model readiness check
router.post('/embed', async (req, res) => {
  try {
    // Check if model is ready before processing
    const health = await embeddingService.getHealth();
    if (!health.ready) {
      return res.status(503).json({ 
        error: 'Embedding model is still loading. Retry in a few seconds.',
        retryAfter: 3,
      });
    }
    // ... rest of handler
```

```typescript
// src/services/treeAwareRetriever.ts — add retry with backoff
async function callEmbeddingService(texts: string[], retries = 2): Promise<number[][]> {
  // ... existing filter logic ...
  
  const response = await fetch(`${API_BASE}/embeddings/embed`, { ... });
  
  if (response.status === 503 && retries > 0) {
    await new Promise(r => setTimeout(r, 2000));
    return callEmbeddingService(texts, retries - 1);
  }
  
  if (!response.ok) {
    throw new Error(`Embedding service error: ${response.status} ${response.statusText}`);
  }
  // ...
}
```

## Part C: Process Improvements

### 1. Pre-Release Smoke Test Checklist

Before any version bump, a human (or a browser-automation test) must verify:

```markdown
## Smoke Test Checklist

- [ ] Open app in browser (fresh state, no cache)
- [ ] Settings → Providers → Connect a provider → Verify "Connected"
- [ ] Settings → MCP tab → Verify it renders without crash
- [ ] Chat → Send "Hello" → Verify response appears
- [ ] Chat → Send follow-up → Verify context is maintained (response references previous message)
- [ ] Chat → Verify markdown renders (bold, code blocks, lists)
- [ ] Chat → Verify pipeline stats appear after response
- [ ] Knowledge → Add a file → Send a question about it → Verify retrieval works
- [ ] Console → No 404s, no 400s, no React errors
- [ ] Export → Export agent → Verify ZIP downloads
```

### 2. Integration Test (Playwright or Cypress)

```typescript
// e2e/chat-flow.spec.ts
test('basic chat conversation maintains history', async ({ page }) => {
  await page.goto('/');
  // Navigate to chat
  await page.fill('[aria-label="Test message"]', 'What is 2+2?');

  await page.waitForSelector('[data-role="assistant"]');
  
  // Send follow-up

  await page.click('[aria-label="Send message"]');
  await page.waitForSelector('[data-role="assistant"]:nth-child(4)');
  
  // Verify the response references the previous question

  expect(response).toContain('2+2'); // LLM should reference the prior question

```

### 3. Ban Unsafe `as` Casts on Message Roles

Add a TypeScript strict pattern:

```typescript
// src/types/messages.ts — Single source of truth
export type MessageRole = 'system' | 'user' | 'assistant';

export interface LLMMessage {
  role: MessageRole;
  content: string;
}
```

Then use this type everywhere. Never use `as` on roles. If TypeScript complains, fix the upstream type, don't cast.

### 4. Agent Output Review Gate

Before merging sub-agent output:
1. Diff review: check for `as` casts, `any` types, `useEffect` with object/array deps
2. If the change touches UI: require a screenshot or browser verification
3. If the agent timed out: flag the output as "unverified" and don't merge without review

### 5. Add an API 404 Catch-All

```typescript
// server/index.ts — after all API routes, before static serving:
app.use('/api', (req, res) => {
  console.warn(`[API 404] ${req.method} ${req.originalUrl}`);
  res.status(404).json({ status: 'error', error: `Not found: ${req.method} ${req.path}` });
});
```

This prevents silent 404s from unmatched API routes and makes debugging trivial.

## Part D: MCP Tab Assessment

### Should the MCP Tab in Settings Be Removed?

**No. Keep it, but refocus it.**

The MCP tab should serve as the **runtime inspector** — showing what's actually running, connection health, tool lists, and logs. The ConnectionPicker handles the *configuration* flow (add/connect new servers). The Settings MCP tab shows the *operational* view (what's connected, what tools are available, recent errors).

**Proposed split:**
- **ConnectionPicker** → "Add & connect MCP servers" (configuration)
- **Settings → MCP** → "Monitor & debug MCP servers" (runtime health, tool inspection, error logs)

This avoids duplication while giving each view a clear purpose.

## Part E: Connector Flow State Machine

### Victor's Request: "Connect" Button → Config → Auto-Add

**State machine:**

```
┌──────────────┐    click "Connect"    ┌──────────────┐    save config    ┌──────────────┐
│ unconfigured │ ──────────────────► │ configuring  │ ──────────────► │  configured  │
│              │                      │ (form open)  │                  │              │
└──────────────┘                      └──────────────┘                  └──────┬───────┘
                                            │                                  │
                                            │ cancel                           │ auto-add to agent
                                            ▼                                  ▼
                                      ┌──────────────┐                  ┌──────────────┐
                                      │ unconfigured │                  │ added-to-    │
                                      │              │                  │ agent        │
                                      └──────────────┘                  └──────┬───────┘
                                                                               │
                                                                   ┌───────────┴───────────┐
                                                                   │                       │
                                                                   ▼                       ▼
                                                            ┌──────────────┐      ┌──────────────┐
                                                            │  connected   │      │ error        │
                                                            │  (badge ✓)   │      │ (retry btn)  │
                                                            └──────────────┘      └──────────────┘
```

**Implementation approach:**

1. Each entry in `MCP_REGISTRY` has a `registryEntry` with auth method, command, args
2. The connector's state lives in `consoleStore.mcpServers[]` with a new `configState` field:
   ```typescript
   type ConnectorConfigState = 'unconfigured' | 'configuring' | 'configured' | 'added-to-agent' | 'connected' | 'error';
   ```
3. The ConnectionPicker's "Connect" button:
   - If `configState === 'unconfigured'` → open inline config form (OAuth URL or command/args)
   - If `configState === 'configured'` → call `/api/mcp/:id/connect`
   - If `configState === 'connected'` → show "Connected ✓" badge + "Edit" + "Remove from agent"
4. First-time config auto-adds to agent: after saving config, call `upsertMcpServer()` and `mcpStore.addServer()` in sequence
5. Already-connected servers show the badge and allow editing (opens Settings → MCP with that server focused)

**Key files to modify:**
- `src/store/mcp-registry.ts` — no changes needed (already has auth methods)
- `src/components/ConnectionPicker.tsx` — add config form states, badges, connect flow
- `src/store/consoleStore.ts` — add `configState` to McpServer type
- `src/store/mcpStore.ts` — ensure `addServer` returns the created server for chaining

**Estimated effort:** 2-3 hours of focused work. This is a UX improvement, not a bug fix — schedule for v1.1.0 after the P0/P1 bugs are fixed.
