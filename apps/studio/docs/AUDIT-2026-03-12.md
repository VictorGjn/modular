## 3. SECURITY AUDIT

### ✅ Good
- **Path traversal protection**: `pathSecurity.ts` validates all file access against allowedDirs
- **Null byte injection blocked**: explicit `\0` check
- **Security headers**: X-Content-Type-Options, X-Frame-Options, Referrer-Policy
- **CORS restricted**: only localhost:5173-5176 origins
- **JSON body limit**: 10MB cap
- **API 404 catch-all**: prevents information leakage on unknown routes
- **Skills audit**: scans for eval(), child_process, unpinned deps, git deps, undocumented skills
- **No raw SQL**: uses sql.js for SQLite with parameterized access

### ⚠️ Moderate Risk
1. **`execFile` in health.ts** — used for `npx skills update --dry-run`. The skill ID comes from URL params (`:id`). While `execFile` is safer than `exec`, the skill ID should be validated against a whitelist pattern (alphanumeric + hyphens only).
   - **File**: `server/routes/health.ts:320`
   - **Fix**: Add `/^[a-z0-9-]+$/i` validation on `req.params.id`

2. **`execFile` in skills-search.ts** — runs `npx clawhub search` with user input from query param. Uses `shell: true` which re-enables injection risk.
   - **File**: `server/routes/skills-search.ts:222`
   - **Fix**: Remove `shell: true`, validate/sanitize search query

3. **CORS in production** — localhost origins only works for dev. Production deploy would need proper origin configuration.

4. **No rate limiting** — API endpoints have no rate limiting. A malicious local process could flood the server.

5. **No auth** — all API endpoints are unauthenticated. Fine for localhost-only tool, but relevant if ever exposed to network.

### 🟢 Low Risk (Acceptable)
- `regex.exec()` calls in pipeline.ts/knowledge.ts — safe, operating on own data
- Dynamic imports in consoleStore (mcpStore reconnect) — controlled, not user-input
